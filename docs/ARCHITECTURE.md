# Архитектура VK Quiz

## Подход

Проект реализован как monorepo modular monolith:

- `apps/web` отвечает за интерфейс, API routes, регистрацию, авторизацию, каталог, solo-прохождение и конструктор квизов.
- `apps/realtime-server` отвечает за live-комнаты и события Socket.IO.
- `packages/db` содержит Prisma-схему и seed.
- `packages/shared` содержит общие события, константы и scoring.

## Основные сценарии

### Регистрация и email verification

1. Пользователь вводит имя, email, пароль и роль.
2. Аккаунт создаётся со статусом `emailVerifiedAt = null`.
3. Система создаёт `VerificationToken`.
4. Email отправляется через Mailpit, SMTP или Resend.
5. Пользователь переходит по `/verify-email?token=...`.
6. Email подтверждается, пользователь получает сессию.

### Публичный каталог

Публичные квизы имеют:

```txt
status = PUBLISHED
visibility = PUBLIC
```

Они отображаются на `/catalog`.

### Приватные квизы

Приватные квизы имеют:

```txt
visibility = PRIVATE
accessCode = SOME-CODE
```

Они не отображаются в каталоге и открываются только по адресу `/catalog/private/[code]`.

### Картинки

У квиза есть `coverImageUrl`, у вопроса есть `imageUrl`. Изображение можно:

- загрузить через `/api/uploads`;
- вставить URL вручную.

Если `coverImageUrl` не задан, UI автоматически показывает фирменную fallback-обложку.

### Solo-прохождение

Для публичных квизов доступен маршрут `/solo/[id]`. Приватный квиз можно пройти самостоятельно по ссылке `/solo/[id]?access=ACCESS_CODE`. Этот режим не требует организатора и live-комнаты.

### Scoring

Система баллов разделяет одиночный и множественный выбор. Для одиночного выбора начисляется полный балл только за правильный вариант. Для multiple-choice используется частичный балл: каждый правильно выбранный вариант добавляет долю очков, каждый лишний вариант вычитает такую же долю, итог не может быть меньше 0. Просроченный ответ в live mode всегда даёт 0 баллов.

### Realtime

Socket.IO использует комнаты вида `room:<code>`. Организатор запускает вопрос, сервер рассылает событие участникам, ответы принимаются только во время активного вопроса.

## Почему Mailpit + SMTP

Mailpit нужен для удобной локальной разработки без настоящей почты. SMTP/Resend нужны для реальной демонстрации и production-сценария. Провайдер переключается через `.env`.
